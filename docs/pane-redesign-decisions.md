# Pane Redesign Decisions

## 1. 意思決定ログ（ステップ4-Aで確定した内容）

ステップ4-B〜4-E は、本ログを実装前提として進める。ここで未定義の仕様を実装中に拡張する場合は、`designing-workspace-ui` の SSoT エスカレーション規律に従い、実装前にユーザー確認を行う。

| 論点 | 決定 | 理由 |
| --- | --- | --- |
| タスクの階層モデル | タスク階層は最大3段階固定（大項目 → 中項目 → 小項目）とし、中項目は任意。DBは単一 `tasks` テーブルに `parentTaskId` と `level` を持たせる。 | ユーザー要望の「Pane2に大項目、場合によって中項目、Pane3に小項目」に合い、既存タスクの移行とCRUDを比較的シンプルに保てるため。 |
| カテゴリの扱い | `Category` はDB/UIともに廃止する。Pane1 はプロジェクト名だけの一階層一覧にする。ステップ2のカテゴリ名編集機能は、ステップ4の再構成で削除対象とする。 | 「プロジェクトカテゴリとプロジェクトがわかりづらい。一階層でよい」というフィードバックに素直に応えるため。 |
| Pane1（プロジェクト名一覧） | Pane1 はフラットなプロジェクト一覧にする。検索を持ち、各行はプロジェクト名を主表示、進捗%と期限バッジを補助表示にする。並び順は未完了優先、期限が近い順にする。 | カテゴリ廃止後は Pane1 がプロジェクト選択の起点になるため、件数が増えても探しやすくする。一方で Pane1 はナビゲーションなので、情報は補助表示に絞る。 |
| Pane2（大項目タスク、場合により中項目タスク） | Pane2 は上にカレンダー、下に大項目タスク一覧を常時表示する。大項目タスク一覧では、選択/展開した大項目に中項目がある場合だけ、中項目を折りたたみ表示する。表示情報はタイトル、進捗%、期限、担当者、完了数。 | カレンダーで期日・完了状態を先に俯瞰し、その下で大項目タスクを判断・選択できる。表示情報は進捗・期限・担当・完了数に絞ることで、優先順位を判断しやすくしつつ情報過多を避ける。 |
| Pane3（小項目タスク＋詳細） | Pane3 は「読む場所」として、小項目タスク一覧を中心に表示する。選択した小項目の詳細は Pane3 内で読み取り表示できるが、編集は従来どおり Pane4 に寄せる。表示情報はタイトル、完了状態、担当者、期限、メモ、親タスクの文脈、進捗%。 | 既存の「Pane3 = 読む場所、Pane4 = 編集の本拠地」という責務分離を維持しながら、「細かい情報まで表示したい」という要望に応えるため。 |
| カレンダービューの配置 | カレンダービューは Pane2 に常時表示する。Pane2 内の配置は、上にカレンダー、下にタスク一覧とする。カレンダーには選択中プロジェクトの大項目・中項目・小項目の期日と完了状態をまとめて表示する。 | 元フィードバックの「2ペイン目にカレンダービュー」に応え、期日と完了状態を先に俯瞰できるようにするため。 |
| 進捗%の算出方法 | 進捗%は小項目完了率ベースにする。小項目は未完了0%、完了100%。中項目・大項目・プロジェクトは配下の小項目タスクの完了数 / 総数で自動計算する。重み付けなし、手入力なし。 | 現行の「進捗率はカラムに持たず都度計算」の方針を維持でき、ユーザーにも説明しやすい。手入力や重み付けによる完了状態とのズレを避けるため。 |
| 完了タスクの並び順UI | Pane2/Pane3 ともに、未完了タスクを上、完了済みタスクを下に分ける。完了済みは折りたたみ可能な「完了済み」セクションに入れ、初期状態では閉じる。 | 「完了したタスクは下に来るように」という要望に応えつつ、完了済みが増えても作業中タスクの視認性を保てるため。Pane2/Pane3 で同じ規律にすると学習コストも下がる。 |
| 既存データの移行 | 既存 `Project` は新モデルでもプロジェクトとして残す。既存 `Task` は小項目タスクへ移行し、各プロジェクトにデフォルト大項目「未分類」を1件作って、その配下に既存タスクをぶら下げる。 | 既存プロジェクトの意味を壊さず、既存タスクも失わずに新しい3階層モデルへ自然に載せ替えられる。既存の軽量タスクを大項目扱いにして粒度を変えてしまうことを避けるため。 |
